Registro de Dueños

🐾

Gestión de Mascotas

Código, nombre, especie, raza y fecha de nacimiento. Cada mascota pertenece exclusivamente a un dueño.

Fecha, hora y motivo de atención. Cada consulta está vinculada a una mascota y es atendida por un único veterinario.

03 · Relaciones y Cardinalidades

Cómo se conectan
las entidades

TIENE 1 : N
DUEÑO
MASCOTA

Un dueño puede tener una o varias mascotas. Cada mascota pertenece a exactamente un dueño.

GENERA 1 : N
MASCOTA
CONSULTA

Una mascota puede tener muchas consultas a lo largo del tiempo. Cada consulta corresponde a una sola mascota.

ATIENDE 1 : N
VETERINARIO
CONSULTA

Un veterinario puede atender muchas consultas. Cada consulta es atendida por un único veterinario.

PRESCRIBE M : N
CONSULTA
TRATAMIENTO

En una consulta se prescriben varios tratamientos, y un mismo tratamiento puede aparecer en varias consultas.

⚠ Nota técnica · Relación M:N → Tabla intermedia

La relación PRESCRIBE entre CONSULTA y TRATAMIENTO requiere la creación de la tabla intermedia CONSULTA_TRATAMIENTO. Esta tabla almacena las FKs de ambas entidades y los atributos propios de la prescripción que no pertenecen a ninguna entidad por separado:

id_consulta FK id_tratamiento FK dosis duracion

Scripts de la base de datos

Creación · Tabla Dueño 
-- Entidad principal: DUEÑO
CREATE TABLE Dueno (
  DNI VARCHAR(8) NOT NULL,
  nombre VARCHAR(100) NOT NULL,
  telefono VARCHAR(20) NULL,
  direccion VARCHAR(200) NULL,
  CONSTRAINT PK_Dueno
    PRIMARY KEY (DNI)
);
Relación M:N · Tabla Intermedia 
-- Tabla intermedia PRESCRIBE
CREATE TABLE Consulta_Tratamiento (
  id_consulta INT NOT NULL,
  id_tratamiento INT NOT NULL,
  dosis VARCHAR(100) NOT NULL,
  duracion VARCHAR(80) NOT NULL,
  CONSTRAINT PK_CT
    PRIMARY KEY
    (id_consulta, id_tratamiento)
);

06 · Conclusiones

Aprendizajes clave
del proyecto

01

El modelo E-R como base del diseño

El modelo Entidad-Relación es fundamental para estructurar la información antes de implementar. Permite identificar datos relevantes, separar responsabilidades y evitar redundancias.

02

Importancia de las cardinalidades

Entender la diferencia entre 1:N y M:N tiene implicaciones directas en la implementación. La relación M:N entre CONSULTA y TRATAMIENTO requirió una tabla intermedia con atributos propios.

03

Del diagrama al SQL

Cada entidad se convierte en tabla, cada atributo en columna, y cada relación en FK o tabla intermedia. Un diagrama bien construido agiliza enormemente la implementación.

04

Competencia profesional aplicable

El modelado E-R no es solo un requisito académico: es una competencia directamente aplicable en ingeniería de sistemas y desarrollo de software real.
